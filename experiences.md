# Experiences

**Tensor indexing (strides)** — `TensorN` now keeps a row-major stride per axis (`m_strides`), recomputed whenever the shape changes. Linear index is `sum_i index[i] * stride[i]` instead of rebuilding tail products with `std::accumulate` on every access. Helpers like slice `assign` reuse cached strides from the source tensor; `linearToRowMajorIndices` takes strides directly. Profiling showed this removed most of the cost that looked like “slow `loopIndices`”: the visitor was fine; repeated \(O(\text{rank}^2)\) index work was not.

**Batch normalization convergence** — Adding `BatchNormLayer` (Conv → BN → ReLU) to the CIFAR-10 momentum SGD demo produced a dramatic improvement in convergence speed. Loss dropped from ~2.0 at epoch 1 without BN to ~0.6 → ~0.4 → ~0.1 across the first three epochs with BN. The mechanism: BN normalizes each channel's activation distribution (zero mean, unit variance) over the batch and spatial dimensions before the nonlinearity, eliminating internal covariate shift. This stabilizes gradients across layers and allows each layer to learn against a consistent input distribution rather than chasing a moving target. In practice the optimizer integrates naturally — `BatchNormLayer` exposes `getWeights()`/`getBias()` as aliases for γ/β, so no optimizer changes were needed. The `momentum` parameter (EMA factor for running mean/variance) controls how quickly the stored inference statistics track the training distribution; `eps` avoids divide-by-zero in low-variance channels.

**Data augmentation vs. overfitting** — Applying random transforms to training inputs (e.g. `make_cifar10_training_augmentation()` in the CUDA demo: horizontal flip, rotation, crop, color jitter, small affine warp, noise, random erasing — OpenCV-backed stages composed in `ImageAugmentationCombinatorial`) materially helps the network **generalize** and **not overfit** the training set. Each epoch sees perturbed versions of the same images, so the model cannot rely on fixed pixel patterns or memorization; it must learn features that survive harmless variation. The tradeoff is extra CPU work to build augmented batches before upload to the GPU, but the regularization effect is real: training accuracy may rise more slowly or stay below a memorizing baseline, while validation or test accuracy is often better relative to training than without augmentation.

**`conv_demo_cuda` network and measured accuracy** — The CIFAR-10 CUDA demo (`run_conv_demo_cuda` / `src/app/conv_demo_cuda.cpp`) uses a deeper, wider CNN on **32×32×3** inputs: **five** convolutional stages in a `ConvolutionalBox` — **32 → 64 → 128 → 128 → 256** output channels (3×3 convs, then a final **1×1** conv at **2×2** spatial size so 3×3 depth can extend without invalid geometry), each followed by batch norm and ReLU, with **max-pool (2×2)** after the first two blocks only. `CudaTensor4<float>` plus **cuDNN** drive conv/BN/pool. The flattened map is **256×2×2 = 1024**; the **dense head** (`CudaTensor<float>`) is **1024 → 512 → ReLU → 256 → ReLU → 10**, with **softmax + cross-entropy** (`SoftmaxCrossEntropyLoss`) and **momentum SGD** (e.g. batch size 1024, L2, shuffled epochs). With the **earlier** two-block (**16/32** channel) two-FC topology, a run reached about **51.89%** training and **49.51%** test accuracy on CIFAR-10; expect different figures after widening and deepening.
