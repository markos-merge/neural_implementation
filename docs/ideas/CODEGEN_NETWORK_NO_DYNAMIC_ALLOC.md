# Idea: Code-Generated Network with No Dynamic Allocation

*This is an exploratory idea, not part of the current implementation plan.*

---

## The idea

**Generate C++ source from a network specification** so that:

1. The **architecture is fixed in code** (specific layers, sizes, order) — no generic "interpreter" that walks a config at runtime.
2. When **compiled**, the binary uses **no dynamic allocation** for the network: all layer state (weights, biases, activations) is **statically or stack-allocated** with fixed dimensions known at codegen time (e.g. `Eigen::Matrix<float, 784, 256>`, `std::array<float, N>`, or a fixed-size `Tensor<Rows, Cols>`).

So: the code generator emits concrete types with fixed dimensions; the compiler lays everything out at compile time. No `malloc`, no `new`, no `std::vector` growth for the network itself — just stack and static storage.

---

## How it could work

1. **Spec** (e.g. YAML, JSON, or a small DSL):
   - Input size, list of layers (linear 256, ReLU, linear 128, ReLU, linear 10), loss, etc.

2. **Code generator** (script or small tool, run at build time):
   - Reads the spec.
   - Emits one or more **autogenerated** C++ files, e.g. `generated_network.hpp` / `generated_network.cpp`.

3. **Generated code**:
   - Uses **fixed-size types** only. For example:
     - Weights: `Eigen::Matrix<float, 784, 256>` for the first layer, `Eigen::Matrix<float, 256, 128>` for the second, etc.
     - Biases: `Eigen::Vector<float, 256>`, etc.
     - Activations: fixed-size buffers per layer (e.g. `std::array<float, 256>` for a hidden layer, or fixed Eigen types).
   - Layer types are templated or specialized on dimensions: `LinearLayer<784, 256>`, or the generator emits a concrete struct that holds only fixed-size members.
   - `forward` and `backward` are inline-friendly calls (or fully unrolled) with no heap allocation. All sizes are compile-time constants.

4. **Build**:
   - CMake runs the code generator, then compiles the generated sources (and your core library with fixed-size Tensor/Layer templates if you go that route).

Result: the **binary** has zero dynamic allocation for the network. Weights, biases, and intermediate activations live in static storage or on the stack. Predictable memory layout, no allocator, no fragmentation — good for embedded, safety-critical, or just maximum predictability.

---

## What “no dynamic allocation” means here

- **No** `malloc` / `new` / `std::vector` (for the network state) in the generated code.
- **No** runtime-sized tensors (e.g. `Tensor(rows, cols)` with runtime values) for weights or activations in the hot path.
- **Yes** to:
  - Fixed-size Eigen matrices/vectors: `Eigen::Matrix<float, I, O>`, `Eigen::RowVectorXf` only if you use fixed `Eigen::Vector<float, N>` for activations.
  - `std::array<float, N>` for buffers.
  - Templates over dimensions: `Tensor<B, I>`, `LinearLayer<I, O>`, so the compiler instantiates exactly the shapes you need.
  - Stack-allocated structs that contain only such fixed-size members.

So “autogenerated files that when compiled have no dynamic allocation” = the generator emits types and code whose sizes are all known at compile time; the compiled binary never allocates heap for the neurons/weights/activations.

---

## Benefits

- **No allocator**: No heap, no fragmentation, no allocator dependency for the network. Useful on bare metal or in constrained environments.
- **Predictable layout**: Addresses and sizes are fixed; easier to reason about cache and memory.
- **Single binary per architecture**: One generated file per spec; each binary is specialized and can be heavily optimized (inlining, constant propagation).
- **Deterministic**: No allocation order or placement surprises at runtime.

---

## Trade-offs

- **Code size / compile time**: Each architecture is a new template instantiation (or new generated structs). Many architectures ⇒ many combinations or many generated files.
- **Stack usage**: If activations are stack-allocated, deep or wide nets can push stack size up. You may need to make some buffers `static` or thread-local instead of on stack.
- **One spec per model**: Changing architecture requires re-running the generator and recompiling. No “load arbitrary config at runtime” without a separate, dynamic path.
- **Core library**: Your current `Tensor` is dynamic (Eigen::Dynamic). You’d need a fixed-size counterpart (e.g. `Tensor<Rows, Cols>` or use Eigen’s fixed types directly) for the generated code to use.

---

## Relation to the current plan

The current plan uses **dynamic** allocation (e.g. `Tensor(rows, cols)`) and a generic, runtime-composed network. This idea is the **opposite**: codegen produces a fully static network with no dynamic allocation. You could add it as an optional path — e.g. a “static” or “codegen” build that emits and compiles fixed-size code for a given spec — while keeping the dynamic library for development and flexibility.
